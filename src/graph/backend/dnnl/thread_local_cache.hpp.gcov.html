      33              : // In multithread scenarios, there are two requests to the kernel's execute()
      34              : // method:
      35              : // 1. To support multithread execution, the kernel instance must be immutable
      36              : //    after its creating, so its execute() method must be const and stateless.
      37              : // 2. To reduce execution overhead, we want to create some mutable resource once
      38              : //    in execute(), cache them and only change few field of them in every
      39              : //    iteration. Those resources may be the dnnl::memory objects or others.
      40              : // Base on above two requests, those mutable resources can't be a part of the
      41              : // kernel instance. So we design this thread_local_cache_t class to cache those
      42              : // mutable resources. Kernel should search the resource it needs in the cache,
      43              : // if found, it should use the found one, otherwise it should create a new one
      44              : // and store it to the cache. At this moment, we observed that those resources
      45              : // will not be shared between threads, so we made the cache be thread local to
      46              : // reduce the search and sync overhead.
      47              : 
      48              : // Note:
      49              : // The shared_ptr of resources in ALL threads are cached in the @global_cache_,
      50              : // which takes the ownership of cached resources. Besides, each thread will use
      51              : // a thread local table to cache the weak_ptr of current thread's resources. The
      52              : // thread local table can be get by using the @get_thread_local_cache() method.
      53              : // - When looking up the cached value, we will only search the thread local
      54              : //   table in thread safe way without lock. If cache hit, we will return the
      55              : //   found value, and the performance should be good.
      56              : // - If cache miss, we need to add a new value to the global table, and add its

      58              : //   lock to protect the global table. The performance should be bad, but cache
      59              : //   miss should be rare.
      60              : // - We can read/write the found resource in each thread without lock, because

      62              : // - If a thread existed, the thread local table will be destroyed, during
      63              : //   which, we will find the share_ptr in global table for each weak_ptr in
      64              : //   thread local table, and release the shared_ptr.

      67              : //   shared ptr in global table will be released. Even though the weak ptr in
